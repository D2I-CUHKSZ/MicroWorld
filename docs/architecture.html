Architecture

System Architecture and Runtime Pipeline

LightWorld already contains the core stages needed for a complete multimodal simulation workflow: ingestion, graph construction, simulation preparation, topology-aware runtime, and report output.

End-to-End Flow

Main pipeline

The current repository structure can be read as one continuous chain, not a disconnected collection of scripts.

1. Multi-modal ingestion

Parse documents, images, and videos into structured event context.

2. Ontology and graph

Generate entity types, edge types, graph chunks, and graph IDs.

3. Simulation preparation

Build entity prompts, platform profiles, social relation graph, and config.

4. Runtime and report

Run platform simulation, collect artifacts, and generate reports.

Runtime Design

What makes the runtime distinctive

Multi-modal first

Images and videos are not treated as optional decoration. They are part of the event package that feeds the simulation pipeline.

SimpleMem-style memory

Memory is stored incrementally and retrieved selectively, which is more scalable than replaying the full conversation history.

Topology-aware unit scheduling

The runtime works on representative units and neighborhoods to reduce unnecessary activation cost.

PPR-driven asymmetry

Personalized PageRank is used to estimate directed influence rather than assuming balanced pairwise effects.

Deployment boundary

GitHub Pages can host the project website and example pages. The actual Flask backend and simulation services still need their own runtime environment if you want an online interactive product.

  • Static project website: GitHub Pages
  • Interactive API service: separate backend deployment
  • Current Pages site: project presentation, guide, and examples
